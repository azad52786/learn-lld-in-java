01 — Foundation

What Are Design Patterns?

"

Design patterns are reusable solutions to commonly occurring problems in software design. They are not finished code — they are templates that describe how to solve a problem in a flexible, proven way.

— Gang of Four (GoF), Design Patterns: Elements of Reusable Object-Oriented Software, 1994

🧠 In Plain English

Imagine you're building a house. Instead of re-inventing how to build a door frame every time, you follow a proven blueprint. Design patterns are those blueprints for code — tried-and-tested ways to structure classes and objects so your code stays clean, flexible, and maintainable.

📖
Origin — The Gang of Four (GoF)

In 1994, four authors — Erich Gamma, Richard Helm, Ralph Johnson, and John Vlissides — published the seminal book that catalogued 23 classic design patterns grouped into 3 categories. Every Java developer should know these.

02 — The Three Categories

3 Types of Design Patterns

The 23 GoF patterns are grouped by the kind of problem they solve. Each type targets a completely different stage of your design.

🏭

Creational

How objects are created

Controls the creation process of objects. Removes tight coupling between what you create and how you create it.

Singleton Factory Method Abstract Factory Builder Prototype
🏗️

Structural

How objects are composed

Deals with assembling objects into larger structures while keeping them flexible and efficient.

Adapter Bridge Composite Decorator Facade Flyweight Proxy
🔄

Behavioral

How objects communicate

Handles communication and responsibility between objects. Defines how they interact and distribute work.

Observer Strategy Command Iterator State Template Method Chain of Resp. Mediator Memento Visitor Interpreter
💡
Easy Way to Remember the 3 Types

Creational = Birth of objects  ·  Structural = Body of the system  ·  Behavioral = Behavior / conversations between objects.

03 — Type One

🏭 Creational Patterns

🏭
Creational Patterns

Object Creation — Controlled & Flexible

Creational patterns abstract the instantiation process. They make a system independent of how its objects are created, composed, and represented. Instead of calling new SomeClass() scattered everywhere, creational patterns give you a structured, centralized way to produce objects.

🔧
Why Use?Avoid scattering new keywords everywhere. Centralise object creation logic.
🔄
FlexibilitySwap the type of object being created without changing the code that uses it.
🧪
TestabilityInject mock objects easily in unit tests instead of hardcoded instances.
♻️
ReuseControl object lifecycle — reuse existing instances (Singleton, Prototype) instead of creating new ones.

Patterns in this group:

  • Singleton Ensures a class has only one instance and provides a global access point to it. Used for shared resources like DB connections, config, logging.
  • Factory Method Defines an interface for creating an object but lets subclasses decide which class to instantiate. Removes the need to bind the code to specific classes.
  • Abstract Factory Creates families of related objects without specifying their concrete classes. Think UI toolkit — same code works for Windows buttons or Mac buttons.
  • Builder Separates the construction of a complex object from its representation. Perfect for objects with many optional parameters (avoids telescoping constructors).
  • Prototype Creates new objects by copying (cloning) an existing object. Useful when object creation is expensive and an existing similar object can be duplicated.
04 — Type Two

🏗️ Structural Patterns

🏗️
Structural Patterns

Object Composition — Building Larger Structures

Structural patterns deal with how classes and objects are composed to form larger structures. They use inheritance and composition to build flexible, efficient structures — like assembling building blocks — without creating rigid, fragile hierarchies.

🔌
CompatibilityMake incompatible interfaces work together without changing their source code (Adapter).
🧩
SimplicityHide complex subsystem details behind a simple interface (Facade).
🎨
ExtensibilityAdd behaviour to objects dynamically without subclassing (Decorator).
🛡️
ControlControl access to objects, add caching, logging, or security around them (Proxy).

Patterns in this group:

  • Adapter Acts as a translator between two incompatible interfaces. Like a power plug adapter — your device doesn't change, the socket doesn't change, the adapter bridges them.
  • Bridge Decouples an abstraction from its implementation so both can vary independently. Separates "what something does" from "how it does it".
  • Composite Treats individual objects and compositions of objects uniformly. Perfect for tree structures — files and folders, UI components, org charts.
  • Decorator Adds new behaviours to objects dynamically by wrapping them. Avoids an explosion of subclasses for every combination of features.
  • Facade Provides a simplified interface to a complex subsystem. One easy entry point hides all the underlying complexity from the client.
  • Flyweight Shares common state between many fine-grained objects to reduce memory usage. Used when you need huge numbers of similar objects (e.g. characters in a text editor).
  • Proxy Provides a surrogate or placeholder for another object to control access to it. Used for lazy loading, caching, access control, and logging.
05 — Type Three

🔄 Behavioral Patterns

🔄
Behavioral Patterns

Object Communication — Who Does What

Behavioral patterns are concerned with algorithms and the assignment of responsibilities between objects. They describe not just patterns of objects or classes but also the patterns of communication between them — who talks to whom, and how work flows.

📢
Loose CouplingObjects communicate without knowing each other's concrete types (Observer, Mediator).
🔀
Interchangeable AlgorithmsSwap algorithms at runtime without changing the client (Strategy).
🎮
State ManagementLet objects change behaviour when their internal state changes (State).
↩️
Undo / HistoryEncapsulate actions as objects to support undo, queuing, and logging (Command, Memento).

Patterns in this group:

  • Observer Defines a one-to-many dependency so that when one object changes state, all its dependents are notified automatically. Used in event systems, UI frameworks, pub-sub.
  • Strategy Defines a family of algorithms, encapsulates each one, and makes them interchangeable. Swap sorting, payment, or discount logic at runtime without if-else chains.
  • Command Encapsulates a request as an object, allowing parameterisation, queuing, logging, and undoable operations. Think of a remote control — each button is a Command object.
  • Iterator Provides a way to sequentially access elements of a collection without exposing its underlying structure. Java's for-each uses this internally.
  • State Allows an object to alter its behaviour when its state changes. The object appears to change its class. Used in vending machines, traffic lights, order workflows.
  • Template Method Defines the skeleton of an algorithm in a base class, letting subclasses override specific steps without changing the overall structure.
  • Chain of Resp. Passes a request along a chain of handlers — each decides to process or pass it on. Used in middleware, filters, approval workflows.
  • Mediator Defines an object that encapsulates how a set of objects interact. Reduces chaotic dependencies — objects don't talk to each other directly, only through the mediator.
  • Memento Captures and externalizes an object's internal state so it can be restored later without violating encapsulation. The undo feature in text editors.
  • Visitor Lets you add new operations to existing objects without modifying them. Separates the algorithm from the object structure it operates on.
  • Interpreter Defines a grammar for a language and provides an interpreter to deal with that grammar. Used in SQL parsers, expression evaluators, scripting engines.

07 — Practical Guidance

🎯 When to Pick Each Type

🏭
Reach for Creational when...

You find yourself writing new SomeClass() in many places · You need only one instance of something (DB, config) · Object construction is complex with many optional parameters · You want to produce different variants of an object.

🏗️
Reach for Structural when...

Two systems have incompatible interfaces and you can't change either · You want to add features to objects without subclassing · You want to simplify a complex API for a client · You need to control or wrap access to an object.

🔄
Reach for Behavioral when...

You have complex conditionals (if-else / switch) based on state or type · Objects need to be notified of changes in other objects · You want to swap algorithms at runtime · You need undo/redo, queuing, or request pipelines.

⚠️
Golden Rule — Patterns are Tools, Not Goals

Never start coding by asking "which pattern should I use?". Start by understanding the problem. Patterns emerge naturally when you recognise a known problem. Forcing a pattern into an unsuitable situation makes code harder — not easier — to understand.
