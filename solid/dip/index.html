🧱 SOLID Principles — LLD with Java

Dependency
Inversion
Principle

The "D" in SOLID. High-level modules must never be enslaved to low-level details — both must bow to abstraction.

Java LLD SOLID Design Patterns Clean Code
01 — Definition

What is DIP?

"

A. High-level modules should not depend on low-level modules. Both should depend on abstractions.

B. Abstractions should not depend on details. Details (concrete implementations) should depend on abstractions.

— Robert C. Martin (Uncle Bob)

🧠 In Plain English

Your high-level business logic (e.g. OrderService) should talk to an interface, not directly to MySQLDatabase or GmailService. Want to swap MySQL for MongoDB tomorrow? Your business logic shouldn't need to change at all.

01

Rule One

High-level modules (business logic) must not instantiate or import concrete low-level classes. Depend on interface or abstract class.

02

Rule Two

Abstractions describe what you need, not how it works. The interface is the socket — the concrete class is just the plug.

02 — Why It Matters

Why Should You Use DIP?

Without DIP With DIP
Business logic is glued to a specific DB, email provider, or payment gateway Business logic talks to an interface — swap the implementation anytime
Unit testing is painful — you can't mock a new MySQL() inside a method Easy to inject a MockDatabase in tests — no real DB needed
Changing a low-level class forces changes throughout the codebase Only the concrete class changes — everything else stays untouched
Tightly coupled — changing vendor = rewriting logic Loosely coupled — add new implementations without touching old code
03 — Bad Example (Violating DIP)

❌ The Problem: Depending on Concrete Classes

An OrderService that hardcodes MySQLDatabase and GmailService directly — the classic DIP violation.

⚠️ BAD — Concrete low-level class
JavaMySQLDatabase.java 
public class MySQLDatabase {
    public void save(String data) {
        System.out.println(
          "MySQL: " + data);
    }
}
⚠️ BAD — Concrete low-level class
JavaGmailService.java
public class GmailService {
    public void sendEmail(String msg) {
        System.out.println(
          "Gmail: " + msg);
    }
}
Java OrderService.java — ❌ VIOLATES DIP 
public class OrderService {

    // ❌ High-level class directly instantiates low-level classes!
    private MySQLDatabase database = new MySQLDatabase();
    private GmailService   emailer  = new GmailService();

    public void placeOrder(String order) {
        database.save(order);      // ❌ tied to MySQL forever
        emailer.sendEmail(order);  // ❌ tied to Gmail forever
        System.out.println("Order placed!");
    }
}
🚨
Why This Breaks Everything

Want to switch MySQL → MongoDB? You must edit OrderService. Want to unit test without a real database? You can't — it's hardwired inside. Every change to a low-level class risks breaking business logic.

↑ High-level depends directly on low-level. Fragile. Untestable. Rigid.

04 — Good Example (Following DIP)

✅ The Fix: Depend on Abstractions

Create interfaces for all external dependencies. Inject them from outside. OrderService only ever knows the interfaces — never the concrete classes.

Java Step 1 — Define Abstractions 
// ✅ Abstraction for storage — doesn't care HOW data is stored
public interface Database {
    void save(String data);
}

// ✅ Abstraction for notifications — doesn't care which provider
public interface NotificationService {
    void send(String message);
}
✅ MySQL — implements Database
JavaMySQLDatabase.java 
public class MySQLDatabase
    implements Database {

  @Override
  public void save(String data) {
    System.out.println(
      "MySQL: " + data);
  }
}
✅ MongoDB — swap with zero changes!
JavaMongoDatabase.java 
public class MongoDatabase
    implements Database {

  @Override
  public void save(String data) {
    System.out.println(
      "MongoDB: " + data);
  }
}
✅ Gmail — implements NotificationService
JavaGmailService.java
public class GmailService
    implements NotificationService {

  @Override
  public void send(String msg) {
    System.out.println(
      "Gmail: " + msg);
  }
}
✅ SMS — plug in without any changes!
JavaSMSService.java
public class SMSService
    implements NotificationService {

  @Override
  public void send(String msg) {
    System.out.println(
      "SMS: " + msg);
  }
}
Java OrderService.java + Main.java — ✅ FOLLOWS DIP 
public class OrderService {

    // ✅ Depends on ABSTRACTIONS — not concrete classes
    private final Database            database;
    private final NotificationService notifier;

    // ✅ Dependencies INJECTED from outside (Constructor Injection)
    public OrderService(Database db, NotificationService notifier) {
        this.database = db;
        this.notifier = notifier;
    }

    public void placeOrder(String order) {
        database.save(order);     // talks to interface ✅
        notifier.send(order);     // talks to interface ✅
        System.out.println("Order placed!");
    }
}

// ── Wiring in Main ─────────────────────────────────────────
public class Main {
    public static void main(String[] args) {

        // Swap implementations with ZERO changes to OrderService
        Database            db      = new MySQLDatabase();  // or MongoDatabase
        NotificationService emailer = new GmailService();   // or SMSService

        OrderService service = new OrderService(db, emailer);
        service.placeOrder("Order #1042");
    }
}
✅
What Changed

OrderService no longer has a single new keyword for its dependencies. To switch MySQL → MongoDB, change one line in Main. Business logic: completely untouched.

OrderService
——depends on——▶
«interface» Database
◀——implemented by——
MySQLDatabase

↑ Both high-level and low-level depend on the abstraction. Stable. Testable. Flexible.

05 — How to Inject

🔌 Three Ways to Inject Dependencies

Java Three Dependency Injection Patterns 
// ── 1. CONSTRUCTOR INJECTION (⭐ Preferred) ──────────────────
public class OrderService {
    private final Database db;

    public OrderService(Database db) { this.db = db; }
    // ✅ Dependency is clear, required, and immutable (final)
}

// ── 2. SETTER INJECTION ──────────────────────────────────────
public class OrderService {
    private Database db;

    public void setDatabase(Database db) { this.db = db; }
    // ⚠️ Optional dependency — object can exist without it
}

// ── 3. INTERFACE INJECTION ───────────────────────────────────
public interface DatabaseInjectable {
    void injectDatabase(Database db);
}
public class OrderService implements DatabaseInjectable {
    private Database db;

    @Override
    public void injectDatabase(Database db) { this.db = db; }
    // 🔵 Rarely used — more common inside frameworks
}
💡
Constructor Injection is King

Always prefer constructor injection. It makes dependencies explicit, enables immutability with final, and makes it impossible to create a broken object that's missing a required dependency.

06 — Watch Out

🚩 Bad Habits to Avoid

  • 🏗️
    Using new inside a class for dependencies Writing private Database db = new MySQLDatabase(); inside a service is the most common DIP violation. The moment you write new ConcreteClass() for a dependency, you've hardwired it forever.
  • 🔗
    Importing concrete low-level classes in high-level modules If your OrderService.java has import com.db.MySQLDatabase at the top — that's a red flag. High-level modules should only import interfaces.
  • 🧪
    Writing untestable code due to hardcoded dependencies If you can't write a unit test without a real database or a real email being sent — your class violates DIP. Testability is a direct proxy for good design.
  • 📦
    Putting business logic inside the concrete implementation Validation and business rules belong in the high-level service — not buried in MySQLDatabase.save(). Low-level classes should be dumb executors, nothing more.
  • ⚙️
    Confusing DIP with Dependency Injection (DI) DIP is the principle (depend on abstractions). DI is one technique to achieve it. You can have DI without DIP (injecting concrete classes directly), but good DI always supports DIP.
  • 🏛️
    Interfaces that leak implementation details An interface with executeSQLQuery(String sql) is not a real abstraction — it's a MySQL wrapper in disguise. A good abstraction says save(Order order) with zero implementation specifics.
💡
The Litmus Test

Can you replace the implementation (DB, email, payment) with a mock in a unit test without changing the class under test? If yes — you're following DIP. If no — time to refactor.

07 — Summary

📋 Quick Cheatsheet

Topic Key Point
What High-level modules must depend on interfaces, not concrete classes
How Define interfaces for all external dependencies. Inject via constructor.
Smell new ConcreteClass() inside a service, or concrete class imports in business logic
Benefit Swap implementations freely, easy unit testing with mocks, loose coupling
DIP vs DI DIP = principle (depend on abstractions). DI = technique (inject from outside).
Best Pattern Constructor injection + final fields — clear, immutable, fail-fast
Works With OCP (new implementations without changing old code), ISP (small interfaces), SRP
🎯
The Golden Rule

If OrderService contains the words MySQL, Gmail, or Stripe anywhere in its file — it has already violated DIP. High-level policy must never know the name of a low-level detail.
