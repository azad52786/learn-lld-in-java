01 — Definition

What is ISP?

"

"A client should not be forced to implement interfaces it does not use. No code should be forced to depend on methods it does not call."

— Robert C. Martin (Uncle Bob)

🧠 In Plain English

If you have a big fat interface with 10 methods, and a class only needs 3 of them — that's a problem. The class is forced to implement 7 useless methods. ISP says: break that big interface into smaller, focused ones.

🏋️ Fat Interface
→
Class forced to implement junk
→
Tight Coupling + Bugs
✂️ Segregated Interfaces
→
Implement only what you need
→
Clean + Flexible Design
02 — Why It Matters

Why Should You Use ISP?

Without ISP With ISP
Classes implement dummy/empty methods just to satisfy an interface Every method in the interface is relevant to the implementor
Changing one method forces recompilation of unrelated classes Changes are isolated — only affected classes need updates
Hard to read: "Why is a Robot implementing eat()?" Clear intent: every class does exactly what it promises
Testing becomes messy with many unused stubs Mocking is focused and clean in unit tests
03 — Bad Example (Violating ISP)

❌ The Problem: Fat Interface

Imagine you're building a system with different types of workers: Human workers and Robot workers. You create one big interface for all of them.

⚠️ BAD — Violates ISP
Java Worker.java 
// ❌ One fat interface — forces ALL workers to implement eat()
public interface Worker {
    void work();
    void eat();      // 🤔 What does a Robot do with this??
    void sleep();     // 🤔 Robots don't sleep either!
}
⚠️ BAD — Robot forced to implement human behaviors
Java Robot.java 
public class Robot implements Worker {

    @Override
    public void work() {
        System.out.println("Robot is working...");
    }

    @Override
    public void eat() {
        // ❌ FORCED to implement this — makes NO sense!
        throw new UnsupportedOperationException("Robots don't eat!");
    }

    @Override
    public void sleep() {
        // ❌ Again — forced junk method!
        throw new UnsupportedOperationException("Robots don't sleep!");
    }
}
🚨
Why This is Dangerous

Throwing UnsupportedOperationException at runtime is a design smell. It means the interface contract is broken. Any code calling worker.eat() on a Robot will crash at runtime, not compile time.

04 — Good Example (Following ISP)

✅ The Fix: Segregated Interfaces

Split the fat interface into focused, role-specific interfaces. Each class implements only what it needs.

Java Interfaces — Workable.java / Eatable.java / Sleepable.java 
// ✅ Small, focused interfaces — each does ONE thing

public interface Workable {
    void work();
}

public interface Eatable {
    void eat();
}

public interface Sleepable {
    void sleep();
}
✅ HumanWorker — implements all 3
JavaHumanWorker.java
public class HumanWorker
    implements Workable, Eatable, Sleepable {

  @Override
  public void work() {
    System.out.println("Human working");
  }

  @Override
  public void eat() {
    System.out.println("Human eating");
  }

  @Override
  public void sleep() {
    System.out.println("Human sleeping");
  }
}
✅ Robot — implements only Workable
JavaRobot.java
// ✅ Robot ONLY implements what it needs!
public class Robot
    implements Workable {

  @Override
  public void work() {
    System.out.println("Robot working 24/7");
  }

  // No eat(), no sleep() — clean!
}
✅
Result: Zero Wasted Methods

Now Robot only has work(). HumanWorker has all three. No forced implementations, no runtime exceptions, no design lies.

05 — Real World Example

🖨️ Printer Example (Closer to Real Code)

A classic ISP example: not all printers can scan, fax, or print in color. Don't lump them all in one interface.

⚠️ BAD — Everything in one interface
JavaMultiFunctionDevice.java
// ❌ A basic printer doesn't fax or scan!
public interface MultiFunctionDevice {
    void print(Document doc);
    void scan(Document doc);    // ❌ basic printers can't scan
    void fax(Document doc);     // ❌ basic printers can't fax
    void copyColor(Document doc); // ❌ B&W printers can't color copy
}
Java ✅ ISP — Segregated printer interfaces 
// ✅ Each interface = one capability
public interface Printable {
    void print(Document doc);
}

public interface Scannable {
    void scan(Document doc);
}

public interface Faxable {
    void fax(Document doc);
}

// Basic printer — only prints
public class BasicPrinter implements Printable {
    @Override
    public void print(Document doc) {
        System.out.println("Printing: " + doc.getName());
    }
}

// All-in-one printer — implements all capabilities
public class AllInOnePrinter implements Printable, Scannable, Faxable {
    @Override
    public void print(Document doc) { /* ... */ }

    @Override
    public void scan(Document doc)  { /* ... */ }

    @Override
    public void fax(Document doc)   { /* ... */ }
}
06 — Watch Out

🚩 Bad Habits to Avoid

  • 🏋️
    The "God Interface" habit Creating one massive interface (e.g., IAnimal with 15 methods) for all subclasses. Not every animal can fly, swim, or roar.
  • 🚫
    Throwing UnsupportedOperationException If you're writing throw new UnsupportedOperationException() in an interface method — that's ISP screaming at you. Break the interface!
  • 🔗
    Coupling unrelated behaviors in one interface Like putting save() and render() in the same interface. Persistence and rendering are different concerns.
  • 📦
    Returning empty/no-op implementations Implementing a method as { } (blank body) just to satisfy the interface is silent ISP violation — harder to detect but equally bad.
  • ⚗️
    Over-segregation (the opposite extreme) Don't go crazy splitting every method into its own interface. If methods are always used together, they belong together. Balance is key.
  • 🧪
    Ignoring ISP in test mocks If your mock classes require stubbing 8 methods to test 1 behavior — your interface is too fat. ISP makes testing naturally cleaner.
💡
The Litmus Test

Ask yourself: "Does EVERY class that implements this interface actually USE every method?" If the answer is no — you need to segregate.

07 — Summary

📋 Quick Cheatsheet

Topic Key Point
What Don't force a class to implement methods it doesn't use
How Split fat interfaces into smaller, role-specific ones
Smell UnsupportedOperationException or blank method bodies in an impl class
Benefit Less coupling, easier testing, cleaner code, fewer surprise bugs
Works with SRP (each interface = one responsibility), DIP, OCP
Avoid Over-segregation — if methods always go together, keep them together
🎯
Remember the 3 Questions

1. Does every implementor use EVERY method?  2. Am I throwing exceptions for "unsupported" methods?  3. Am I testing with too many stubs? If yes → apply ISP.
